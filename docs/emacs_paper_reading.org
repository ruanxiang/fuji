(defun my-gptel-pdf-screenshot-query ()
  "Take a screenshot of a PDF region and query gptel about it."
  (interactive)
  ;; 定义截图保存路径
  (let* ((filename (format "/tmp/emacs-gptel-screenshot-%s.png" (format-time-string "%Y%m%d%H%M%S")))
         ;; -s: select region, -u: hide cursor
         (maim-cmd (format "maim -s -u %s" (shell-quote-argument filename))))
    ;; 最小化Emacs窗口以进行截图
    (x-iconify-frame)
    (unwind-protect
        ;; 执行截图命令
        (if (zerop (call-process-shell-command maim-cmd nil 0))

*** 1. PDF 直接转为多页图像交给 gptel

这个方法理论上是可行的，但实践中会遇到*三个主要瓶颈*，导致它通常不是最佳选择。

-   *技术实现*：
    我们可以用 =poppler-utils= 包里的 =pdftoppm= 命令轻松地将一个 PDF 转换成一堆图片。
    #+begin_src bash
    # 将 my-paper.pdf 转换成 my-paper-01.png, my-paper-02.png ...
    pdftoppm -png my-paper.pdf my-paper
    #+end_src
    然后，你可以将这些图片文件批量拖入或通过 =gptel-insert-image= 插入到 =gptel= buffer 中。

-   *为什么不推荐这样做（关键瓶颈）*：

    1.  *上下文窗口限制 (Context Window Limit)*：这是最大的问题。LLM 的上下文窗口是按 "token" 计算的。一张高分辨率图片会消耗掉*巨量*的 token。例如，一张图片可能就相当于数千甚至上万个文本 token。一篇15页的论文就是15张大图片，几乎*百分之百会超出*当前任何主流模型（包括 GPT-4o）的单次请求上下文限制。模型无法一次性“看到”整篇论文，也就无法进行全文的总结和前后文关联。

    2.  *高昂的成本 (High Cost)*：视觉模型的计费方式中，图片 token 的价格远高于文本 token。将整篇论文作为图片处理，其 API 调用成本会比处理纯文本高出几个数量级。

    3.  *交互性差 (Poor Interactivity)*：模型返回的答案是基于它对图片的“视觉理解”。你无法方便地从论文原文中复制粘贴一小段文字来追问，因为你面前的只是一张张图片。这破坏了 Emacs 以文本为中心的流畅交互体验。

*结论*：这个方法只适用于非常短的文档（比如1-3页的海报或摘要），对于一篇完整的学术论文来说，成本和技术限制都太大了。
